# Operator Invocation

## Usage Instructions

- Prerequisites: Before invoking project operators, please refer to [Environment Deployment](../install/quick_install.md) to complete basic environment setup and source code download.
- Operator Scope: The scope of project operators that can be invoked refers to [Operator List](../../zh/op_list.md). The corresponding aclnn interfaces for operators refer to [aclnn List](../../zh/op_api_list.md).
- build.sh: Operator invocation is implemented based on build.sh. Command options can be viewed through `bash build.sh --help`. Parameter introduction refers to [build Parameter Description](../install/build.md).
- Operator Invocation Methods: Currently operators support aclnn API (recommended) and graph mode invocation. Invocation principles and processes refer to [Operator Invocation Methods](op_invocation.md).

If your compilation environment can access the network, please refer to [Online Compilation](#online-compilation). The compilation script will automatically download third-party software online. Otherwise, please refer to [Offline Compilation](#offline-compilation) to manually download third-party software.

After preparing open source third-party software, you can use the following compilation methods. Please choose according to your needs:

- **Custom Operator Package**:

  The package generated by selecting partial operators for compilation is called custom operator package. It acts on CANN package in **mounting** mode without changing original package content. The generated custom operator package has higher priority than original CANN package. This package supports aclnn and graph mode invocation of AI Core and AI CPU operators.

- **ops-cv Package**:

  The package generated by selecting the entire project for compilation is called ops-cv package. It can **completely replace** the corresponding part of CANN package. This package supports aclnn and graph mode invocation of AI Core operators.

- **ops-cv Static Library**:

  > Note: If you need to **perform secondary release based on this project** and have **requirements for software package size**, it is recommended to use static library compilation. This library can link your application development program, only retaining operators needed by business, thus achieving software minimization deployment.

  Indicates compiling the entire project into a static library file, including libcann_cv_static.a and aclnn interface header files. This package only supports aclnn invocation of AI Core operators.

### Online Compilation

#### Custom Operator Package

1. **Compile Custom Operator Package**

    Enter the project root directory and execute the following compilation command:

    ```bash
    bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}] [-j${n}]
    # Example: Compile grid_sample operator
    # bash build.sh --pkg --soc=ascend910b --ops=grid_sample -j16
    # Compile user operators under experimental contribution directory (taking grid_sample operator as example, please refer to actual contributed operators during compilation)
    # bash build.sh --pkg --experimental --soc=ascend910b --ops=grid_sample
    ```

    - --soc: $\{soc\_version\} represents NPU model. Atlas A2 training series products/Atlas A2 inference series products use "ascend910b" (default), Atlas A3 training series products/Atlas A3 inference series products use "ascend910_93", Ascend 950PR/Ascend 950DT products use "ascend950".
    - --vendor_name (optional): $\{vendor\_name\} represents the constructed custom operator package name, default name is custom.
    - --ops (optional): $\{op\_list\} represents operators to be compiled. When not specified, all operators are compiled by default. Format like "grid_sample,iou_v2,...", multiple operators separated by comma ",".
    - --experimental (optional): Indicates compiling operators saved by users under experimental contribution directory.
    - -j (optional): Specify compilation thread count to speed up compilation.

    If neither $\{vendor\_name\} nor $\{op\_list\} is passed, the compiled package is ops-cv package. If compiling custom operator package for all operators, need to pass $\{vendor\_name\}. When the following message is prompted, compilation is successful.

    ```bash
    Self-extractable archive "cann-ops-cv-${vendor_name}_linux-${arch}.run" successfully created.
    ```

    After successful compilation, the run package is stored in the build_out directory under the project root directory.

2. **Install Custom Operator Package**

    ```bash
    ./build_out/cann-ops-cv-${vendor_name}_linux-${arch}.run
    ```

    The custom operator package installation path is ```${ASCEND_HOME_PATH}/opp/vendors```. $\{ASCEND\_HOME\_PATH\} has been configured through environment variable, representing CANN toolkit package installation path, generally $\{install\_path\}/cann.

    > Note: If the operator package installation directory is specified through --install-path parameter during deployment, before using custom operators, need to execute source $\{install-path\}/vendors/${vendor_name}/bin/set_env.bash command. The set_env.bash script appends the custom operator package installation path to environment variable ASCEND_CUSTOM_OPP_PATH, making custom operators effective in current environment.

3. **(Optional) Uninstall Custom Operator Package**

    After custom operator package installation, `uninstall.sh` will be generated in ```${ASCEND_HOME_PATH}/opp/vendors/${vendor_name}_cv/scripts``` directory. You can uninstall custom operator package through this script with the following command:

    ```bash
    bash ${ASCEND_HOME_PATH}/opp/vendors/${vendor_name}_cv/scripts/uninstall.sh
    ```

#### ops-cv Package

1. **Compile ops-cv Package**

    Enter the project root directory and execute the following compilation command:

    ```bash
    # Compile all operators except experimental directory
    bash build.sh --pkg --soc=${soc_version} [-j${n}]
    # Compile all operators under experimental directory
    # bash build.sh --pkg --experimental --soc=${soc_version} [-j${n}]
    ```

    - --soc: $\{soc\_version\} represents NPU model. Atlas A2 training series products/Atlas A2 inference series products use "ascend910b" (default), Atlas A3 training series products/Atlas A3 inference series products use "ascend910_93", Ascend 950PR/Ascend 950DT products use "ascend950".
    - --experimental (optional): Indicates compiling operators saved by users under experimental directory.
    - -j (optional): Specify compilation thread count to speed up compilation.

    If the following message is prompted, compilation is successful.

    ```bash
    Self-extractable archive "cann-${soc_name}-ops-cv_${cann_version}_linux-${arch}.run" successfully created.
    ```

    $\{soc\_name\} represents NPU model name, which is the remaining content after removing "ascend" from $\{soc\_version\}. After successful compilation, the run package is stored in build_out directory.

2. **Install ops-cv Package**

    ```bash
    # Installation command
    ./build_out/cann-${soc_name}-ops-cv_${cann_version}_linux-${arch}.run --full --install-path=${install_path}
    ```

    $\{install\_path\}: Represents specified installation path, needs to be installed in the same path as toolkit package, default installation is in `/usr/local/Ascend` directory.

3. **(Optional) Uninstall ops-cv Package**

    ```bash
    # Uninstall command
    ./${install_path}/cann/share/info/ops_cv/script/uninstall.sh
    ```

#### ops-cv Static Library

> Note: Static library only supports Atlas A2 and Atlas A3 series products. Experimental operators do not support using static library temporarily.

1. **Compile ops-cv Static Library**

    Enter the project root directory and execute the following compilation command:

    ```bash
    bash build.sh --pkg --static --soc=${soc_version} [-j${n}]
    ```

    - --soc: $\{soc\_version\} represents NPU model. Atlas A2 series products use "ascend910b" (default), Atlas A3 series products use "ascend910_93".
    - -j (optional): Specify compilation thread count to speed up compilation.
    If the following message is prompted, compilation and compression are successful.

    ```bash
    [SUCCESS] Build static lib success!
    Successfully created compressed package: ${repo_path}/build_out/cann-${soc_name}-ops-cv-static_${cann_version}_linux-${arch}.tar.gz
    ```

    $\{repo\_path\} represents project root directory, $\{soc\_name\} represents NPU model name, which is the remaining content after removing "ascend" from $\{soc\_version\}. After successful compilation, the compressed package is stored in build_out directory.

2. **Extract ops-cv Static Library**

    Enter build_out directory and execute extraction command:

    ```bash
    tar -zxvf ./cann-${soc_name}-ops-cv-static_${cann_version}_linux-${arch}.tar.gz -C ${static_lib_path}
    ```

    $\{static\_lib\_path\} represents static library extraction path. The directory structure after extraction is as follows:

    ```cpp
    ├── cann-${soc_name}-ops-cv-static_${cann_version}_linux-${arch}
    │   ├── lib64
    │   │   ├── libcann_cv_static.a                 # Static library file
    │   └── include
    |       ├── ...                                 # aclnn interface header files
    ```

### Offline Compilation

If compiling in an environment without internet connection, you need to prepare dependent third-party software in advance, then perform source code compilation. The specific process is as follows:

1. **Check if Basic Environment is Complete**

    Please ensure basic environment setup has been completed according to [Environment Deployment](../install/quick_install.md), including CANN package installation, source code download, etc.

    - In networked environment, enter [this project homepage](https://gitcode.com/cann/ops-cv), through `Download ZIP` or `clone` button, complete source code download according to instructions.
    - Connect to offline environment, upload source code to your specified directory. If downloaded source code compressed package, please extract first.

2. **Download Third-Party Software Dependencies**

    Download third-party software in advance in networked environment. Currently there are the following methods, please choose according to your needs:

    - Method 1: Manually download according to the table provided in [Third-Party Software Dependencies](#third-party-software-dependencies). If downloading from other addresses, please ensure version numbers are consistent.

    - Method 2: One-click download through [third_lib_download.py](../../../scripts/tools/third_lib_download.py) script. This script is in this project `scripts/tools/` directory. Download this script and execute the following command:

        ```bash
        python ${scripts_dir}/third_lib_download.py
        ```

    $\{scripts\_dir\} represents script storage path. Downloaded third-party software packages are stored in the directory where current script is located by default.

3. **Compile Operator Package**

    Upload downloaded third-party software to offline environment, can be stored in `third_party` directory or custom directory. **The former is recommended, its compilation command is consistent with the command in online compilation scenario.**

    - **third\_party directory** (recommended)

        Please create `third_party` directory in this project root directory (if exists, no need to create), copy third-party software to this specified directory. At this time, compilation command is consistent with online compilation command, specifically refer to [Online Compilation](#online-compilation).

    - **Custom directory**

        Create `${cann_3rd_lib_path}` directory at any location in offline environment, copy third-party software to this directory. Please ensure this directory has access permission.

        ```bash
        mkdir -p ${cann_3rd_lib_path}
        ```

        At this time, compilation command needs to add `--cann_3rd_lib_path=${cann_3rd_lib_path}` on top of online compilation command to specify third-party software location. Assuming storage path is `/path/cann_3rd_lib_path`, commands for different compilation methods are as follows:

        - Custom operator package

            ```bash
            bash build.sh --pkg --soc=${soc_version} [--vendor_name=${vendor_name}] [--ops=${op_list}] --cann_3rd_lib_path=${cann_3rd_lib_path}
            # Example: Compile grid_sample operator
            # bash build.sh --pkg --soc=ascend910b --ops=grid_sample -j16 --cann_3rd_lib_path=/path/cann_3rd_lib_path
            ```

        - ops-cv full package

            ```bash
            bash build.sh --pkg --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
            # bash build.sh --pkg --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
            ```

        - ops-cv static library

            ```bash
            bash build.sh --pkg --static --soc=${soc_version} --cann_3rd_lib_path=${cann_3rd_lib_path}
            # bash build.sh --pkg --static --soc=ascend910b --cann_3rd_lib_path=/path/cann_3rd_lib_path
            ```

4. **Install/Uninstall Operator Package**

    Operator package results compiled in offline and online scenarios are the same, stored in build_out directory under project root directory by default. Installation and uninstallation operation commands are also the same, specifically refer to [Online Compilation](#online-compilation).

## Local Verification

Execute operators and UT test cases through build.sh in project root directory. Currently operators support aclnn API (recommended) and graph mode invocation. Invocation principles and processes refer to [Operator Invocation Methods](op_invocation.md).

### Execute Operator Examples

> **Note**: Ascend 950PR product uses simulation to execute operator examples, please refer to [Simulation Guide](../debug/op_debug_prof.md#method-2-for-ascend-950pr).

- Execute operator examples based on **custom operator package**. After package installation, execute the following command:

    ```bash
    bash build.sh --run_example ${op} ${mode} ${pkg_mode} [--vendor_name=${vendor_name}] [--soc=${soc_version}] [--experimental]
    # Example: Execute grid_sample operator example
    # bash build.sh --run_example grid_sample eager cust --vendor_name=custom
    # Example: Execute grid_sample operator experimental
    # bash build.sh --experimental --run_example grid_sample eager cust --vendor_name=custom
    ```

    - $\{op\}: Represents operator to be executed, operator name in lowercase underscore form, such as grid_sample.
    - $\{mode\}: Represents execution mode, currently supports eager (aclnn invocation), graph (graph mode invocation).
    - $\{pkg_mode\}: Represents package mode, currently only supports cust, which is custom operator package.
    - $\{vendor\_name\} (optional): Consistent with constructed custom operator package setting, default name is custom.
    - $\{soc_version\} (optional): Represents NPU model.
    - $\{experimental\} (optional): Represents executing operators saved by users in experimental contribution directory.

    Note: When $\{mode\} is graph, do not specify $\{pkg_mode\} and $\{vendor\_name\}

- Execute operator examples based on **ops-cv package**. After installation, execute the following command:

    ```bash
    bash build.sh --run_example ${op} ${mode} [--soc=${soc_version}]
    # Example: Execute grid_sample operator example
    # bash build.sh --run_example grid_sample eager
    ```

    - $\{op\}: Represents operator to be executed, operator name in lowercase underscore form, such as grid_sample.
    - $\{mode\}: Represents operator execution mode, currently supports eager (aclnn invocation), graph (graph mode invocation).
    - $\{soc_version\} (optional): Represents NPU model.

- Execute operator examples based on **ops-cv static library**:

    1. **Prerequisites**

        ops-cv static library depends on ops-legacy static library. Prepare the above static libraries, extract and move all lib64 and include directories to unified directory $\{static\_lib\_path\}.

        > Note: ops-legacy static library ```cann-${soc_name}-ops-legacy-static_${cann_version}_linux-${arch}.tar.gz``` can be obtained by clicking [Download Link](https://ascend.devcloud.huaweicloud.com/artifactory/cann-run-mirror/software/master). ops-cv static library does not provide software package temporarily, please generate through local compilation.

    2. **Create run.sh**

        Create run.sh file in the same directory as operator to be executed `examples/test_aclnn_${op_name}.cpp`.

        Taking grid_sample operator executing test_aclnn_grid_sample2_d.cpp and test_aclnn_grid_sample3_d as examples:

        ```bash
        # Static library file path
        static_lib_path=""

        # Environment variable takes effect
        if [ -n "$ASCEND_INSTALL_PATH" ]; then
            _ASCEND_INSTALL_PATH=$ASCEND_INSTALL_PATH
        elif [ -n "$ASCEND_HOME_PATH" ]; then
            _ASCEND_INSTALL_PATH=$ASCEND_HOME_PATH
        else
            _ASCEND_INSTALL_PATH="/usr/local/Ascend/cann"
        fi

        source ${_ASCEND_INSTALL_PATH}/bin/setenv.bash

        # Compile executable file
        g++ test_aclnn_grid_sample2_d.cpp \
        -I ${static_lib_path}/include \
        -L ${static_lib_path}/lib64 \
        -I ${_ASCEND_INSTALL_PATH}/include \
        -I ${_ASCEND_INSTALL_PATH}/include/aclnnop \
        -L ${_ASCEND_INSTALL_PATH}/lib64 \
        -Wl,--allow-multiple-definition \
        -Wl,--start-group -lcann_cv_static -lcann_math_static -lcann_legacy_static -Wl,--end-group -lgraph -lgraph_base \
        -lpthread -lmmpa -lmetadef -lascendalog -lregister -lopp_registry -lops_base -lascendcl -ltiling_api -lplatform \
        -ldl -lc_sec -lnnopbase -lruntime -lerror_manager -lunified_dlog \
        -o test_aclnn_grid_sample2_d   # Replace with actual operator executable file name

        g++ test_aclnn_grid_sample3_d.cpp \
        -I ${static_lib_path}/include \
        -L ${static_lib_path}/lib64 \
        -I ${_ASCEND_INSTALL_PATH}/include \
        -I ${_ASCEND_INSTALL_PATH}/include/aclnnop \
        -L ${_ASCEND_INSTALL_PATH}/lib64 \
        -Wl,--allow-multiple-definition \
        -Wl,--start-group -lcann_cv_static -lcann_math_static -lcann_legacy_static -Wl,--end-group -lgraph -lgraph_base \
        -lpthread -lmmpa -lmetadef -lascendalog -lregister -lopp_registry -lops_base -lascendcl -ltiling_api -lplatform \
        -ldl -lc_sec -lnnopbase -lruntime -lerror_manager -lunified_dlog \
        -o test_aclnn_grid_sample3_d   # Replace with actual operator executable file name

        # Execute program
        ./test_aclnn_grid_sample2_d
        ./test_aclnn_grid_sample3_d
        ```

        $\{static\_lib\_path\} represents static library unified placement path; $\{ASCEND\_INSTALL\_PATH\} has been configured through environment variable, representing CANN toolkit package installation path; final executable file name please replace with **actual operator executable file name**.

        Among them, lcann\_cv\_static and lcann\_legacy\_static represent static library files that operators depend on, obtained from static library unified placement path $\{static\_lib\_path\};
        lgraph, lmetadef, etc. represent underlying library files that operators depend on, can be obtained in CANN toolkit package.

    3. **Execute run.sh**

        ```bash
        bash run.sh
        ```

### Execute Operator UT

> Note: Executing UT test cases depends on googletest unit testing framework. For detailed introduction, refer to [googletest official website](https://google.github.io/googletest/advanced.html#running-a-subset-of-the-tests).

```bash
# Install test related requirements.txt dependencies under root directory
pip3 install -r tests/requirements.txt
# Method 1: Compile and execute specified operator and corresponding function UT test cases (choose one)
bash build.sh -u --[opapi|ophost|opkernel] --ops=iou_v2
# Method 2: Compile and execute all UT test cases
# bash build.sh -u
# Method 3: Compile all UT test cases but do not execute
# bash build.sh -u --noexec
# Method 4: Compile and execute corresponding function UT test cases (choose one)
# bash build.sh -u --[opapi|ophost|opkernel]
# Method 5: Compile corresponding function UT test cases but do not execute (choose one)
# bash build.sh -u --noexec --[opapi|ophost|opkernel]
# Method 6: Specify soc compilation when executing UT test cases
# bash build.sh -u --[opapi|ophost|opkernel] [--soc=${soc_version}]
```

Assuming verifying whether ophost function is normal, execute the following command:

```bash
bash build.sh -u --ophost
```

After execution completes, the following content appears, indicating successful execution.

```bash
Global Environment TearDown
[==========] ${n} tests from ${m} test suites ran. (${x} ms total)
[  PASSED  ] ${n} tests.
[100%] Built target cv_op_host_ut
```

$\{n\} represents n test cases executed, $\{m\} represents m tests, $\{x\} represents time consumed executing test cases, in milliseconds.
